io.cpp — vstup/výstup¶

Zápis výsledků do souborů: VTK pro ParaView, textová data pro Python, export matic. Ukazuje práci se souborovými proudy a proč je I/O oddělené od fyziky.

Proč samostatný modul¶

Fyzika (Simple.cpp) a zápis souborů jsou oddělené odpovědnosti. Solver počítá pole, io:: je serializuje. Výhody:

  • Solver nezávisí na formátu výstupu — můžu přidat nový formát bez zásahu do fyziky.
  • Funkce v namespace io jsou volné funkce (ne metody solveru) — berou data jako const reference, nic nemění.
namespace io {
void writeVTK(const std::string& path, const Mesh& m,
              const volVector& U, const volScalar& p, const volScalar& res);
...
}

VTK výstup¶

void writeVTK(const std::string& path, const Mesh& m, ...) {
    std::ofstream f(path);
    f << "# vtk DataFile Version 3.0\n3D channel cube flow\nASCII\n"
         "DATASET STRUCTURED_POINTS\n";
    f << "DIMENSIONS " << m.nx << " " << m.ny << " " << m.nz << "\n";
    f << "SPACING " << m.dx << " " << m.dy << " " << m.dz << "\n";
    f << "POINT_DATA " << m.N() << "\n";

    f << "VECTORS velocity double\n";
    for(int c=0;c<m.N();++c) f<<U[c].x<<" "<<U[c].y<<" "<<U[c].z<<"\n";
    // + SCALARS pressure, solid, continuity_res
}

std::ofstream a RAII¶

std::ofstream f(path);   // otevře soubor
// ... zápis ...
// soubor se zavře AUTOMATICKY při zániku f (konec funkce)

ofstream je RAII obal nad souborem: konstruktor otevře, destruktor zavře. Žádné ruční fclose, žádný únik při výjimce. To je jeden z hlavních důvodů, proč C++ I/O proudy preferovat před C FILE*.

Formát STRUCTURED_POINTS¶

Protože síť je uniformní kartézská, VTK stačí říct rozměry, počátek a krok (SPACING) — body se negenerují explicitně. Kompaktní a ParaView ho přímo chápe. Pole solid se exportuje taky, aby šla krychle v ParaView odfiltrovat.

Textový výstup pro Python¶

void writeFields(const std::string& path, const Mesh& m, ...) {
    std::ofstream f(path);
    f << "# i j k x y z ux uy uz p res solid\n";
    for(/* všechny buňky */){
        f<<i<<" "<<j<<" "<<k<<" "<<(i+0.5)*m.dx<<" "/* ... */;
    }
}

Prostý sloupcový textový formát, který numpy.loadtxt načte jedním voláním. Komentář # na začátku přeskočí. Souřadnice (i+0.5)*dx jsou středy buněk.

Export matic¶

void writePMatrix(const std::string& path, const Mesh& m,
                  const fvMatrix<double>& mat);
void writeUMatrix(const std::string& path, const Mesh& m,
                  const fvMatrix<Vec3>& mat);

Dvě funkce pro dva typy matic — vypisují všech 7 koeficientů na buňku. Slouží vizualizaci sparsity patternu (plot_matrix.py). writeUMatrix navíc rozepíše src na tři složky (src_x, src_y, src_z).

std::FILE* pro živý zápis reziduí¶

V Simple.cpp se naopak pro průběžný zápis convergence.dat použil C-style FILE*:

std::FILE* convFile = std::fopen(convergencePath.c_str(), "w");
std::fprintf(convFile, "%6d  %.6e  %.6e  %.6e\n", it, ...);
std::fflush(convFile);   // ← vyprázdní buffer HNED, vidíme progress živě

Proč tady FILE* a ne ofstream? Kvůli explicitnímu fflush po každé iteraci — chceme reziduum vidět živě, ne až po doběhnutí (kdy se buffer vyprázdní sám). S ofstream by se to dělalo přes std::endl nebo flush(), FILE* je tu přímočařejší.

Dvě I/O cesty, každá za jiným účelem

ofstream (RAII, bezpečné) pro finální výpis polí na konci. FILE* + fflush pro průběžný živý zápis reziduí během výpočtu. Volba nástroje podle potřeby.
